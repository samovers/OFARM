# CP15 Phase 5 — Machine-Contract Plan
Status: CP15 Phase 5 draft machine-contract plan.
Posture: draft/non-default only; no current/default promotion; no production software-delivery, model-deployment, cybersecurity, or autonomous release-readiness claim.
## Boundary invariant
Generated software, generated adapters, generated mappings, model improvements, prompt/workflow changes, build success, test success, security scan completion, conformance run completion, capability declaration, canary success, deployment telemetry, or agent tool success do not create deployment authority, runtime authority, current/default promotion, mission authority, Compliance Twin fact, or production readiness by themselves.
## Contract families
### SoftwareDeliveryBoundary
- Purpose: Declares the CP15 software-delivery boundary and non-bypass posture.
- Required fields: `boundaryState, scopeRefs, nonBypassClauses, explicitDeferrals, applicableGateRefs, doesNotCreateDeploymentAuthority, doesNotCreateRuntimeAuthority, doesNotPromoteCurrentDefault`.
- Optional fields: `none`.
- Validation rules: Requires non-bypass clauses and explicit deferrals.
- Lifecycle: DRAFT -> ACTIVE_CANDIDATE -> ACCEPTED_NON_DEFAULT -> SUPERSEDED.
- Authority source: authority is always external to the artifact through AuthorityDecisionTrace, approval/review references, or existing OFARM authority law; the artifact itself is not an AuthorityGrant.
- Relation to events: CP15 events should record generation, build, scan, conformance, authorization, release, deployment, rollback, telemetry, and incident transitions without converting them into runtime authority by event presence alone.
- Relation to current-state materialisation: CP15 artifacts may reference current-state materialisations only as evidence basis; they do not create current state or current/default promotion.
- Relation to Advisory Twin: Drafts, candidates, evaluations, telemetry, and model evidence are Advisory Twin material unless separately bridged through existing OFARM gates.
- Relation to Compliance Twin: No CP15 object creates Compliance Twin fact by itself; compliance-relevant outcomes require existing review, evidence, authority, and output gates.
- Relation to CP11 sustainability gates: Charter-sensitive deployments require CP11 gate refs, sustainability-output qualifications, and claim-basis preservation where relevant.
- Relation to CP12 mission/adapter gates: Mission, robot, command, or adapter-related deployments require CP12 mission/adapter gate refs and must not create mission authority.
- Relation to CP13 learning/model-output gates: Model or learning-derived deployments must reference CP13 learning-output qualification and cannot treat learning output as deployment authority.
- Relation to CP14 training/federated/model-improvement gates: Training/federated/model-improvement related deployments must reference CP14 training-use and model-improvement boundary evidence.
- Relation to agent run traces and tool manifests: Generated or agent-managed artifacts must link to agent run traces/tool manifests where applicable; tool success is not authority.
- Relation to security/conformance evidence: Security/conformance evidence is required by candidate class and cannot by itself create deployment authority.
- Relation to deployment authorization, runtime-surface binding, rollback, telemetry, and incidents: Deployment requires DeploymentAuthorization, DeploymentPromotionDecision, ReleaseBundle, RuntimeSurfaceReleaseBinding, rollback posture, telemetry, and incidents as applicable.
- Conformance tests: validate required evidence; reject missing authority where state implies approval; reject any attempt to use SoftwareDeliveryBoundary as deployment authority without CP15 authorisation.
- Example: `04_implementation_and_conformance/examples_and_fixtures/examples/machine_contracts/agentic_software_delivery_model_deployment/OFARM_SoftwareDeliveryBoundary_example_v0_1.json`.

- Relation to current-state materialisation: CP15 artifacts may reference current-state materialisations only as evidence basis; they do not create current state or current/default promotion.
- Relation to Advisory Twin: Drafts, candidates, evaluations, telemetry, and model evidence are Advisory Twin material unless separately bridged through existing OFARM gates.
- Relation to Compliance Twin: No CP15 object creates Compliance Twin fact by itself; compliance-relevant outcomes require existing review, evidence, authority, and output gates.
- Relation to CP11 sustainability gates: Charter-sensitive deployments require CP11 gate refs, sustainability-output qualifications, and claim-basis preservation where relevant.
- Relation to CP12 mission/adapter gates: Mission, robot, command, or adapter-related deployments require CP12 mission/adapter gate refs and must not create mission authority.
- Relation to CP13 learning/model-output gates: Model or learning-derived deployments must reference CP13 learning-output qualification and cannot treat learning output as deployment authority.
- Relation to CP14 training/federated/model-improvement gates: Training/federated/model-improvement related deployments must reference CP14 training-use and model-improvement boundary evidence.
- Relation to agent run traces and tool manifests: Generated or agent-managed artifacts must link to agent run traces/tool manifests where applicable; tool success is not authority.
- Relation to security/conformance evidence: Security/conformance evidence is required by candidate class and cannot by itself create deployment authority.
- Relation to deployment authorization, runtime-surface binding, rollback, telemetry, and incidents: Deployment requires DeploymentAuthorization, DeploymentPromotionDecision, ReleaseBundle, RuntimeSurfaceReleaseBinding, rollback posture, telemetry, and incidents as applicable.
- Conformance tests: validate required evidence; reject missing authority where state implies approval; reject any attempt to use DeploymentOutputQualification as deployment authority without CP15 authorisation.
- Example: `04_implementation_and_conformance/examples_and_fixtures/examples/machine_contracts/agentic_software_delivery_model_deployment/OFARM_DeploymentOutputQualification_example_v0_1.json`.

## Phase 5 limitations
- The schemas are initial draft/non-default shapes, not hardened final contracts.
- Conditional and cross-record hardening is expected in Phase 6 hostile review and Phase 6.1 remediation.
- CP15 does not create production CI/CD, generic MLOps, legal/security certification, robot mission, farm-to-farm intelligence, or current/default schema promotion law.
