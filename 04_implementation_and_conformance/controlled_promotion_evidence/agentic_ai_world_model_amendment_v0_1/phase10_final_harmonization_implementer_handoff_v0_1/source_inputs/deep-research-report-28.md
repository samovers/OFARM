# OFARM Agentic Amendment Research Report

## Governance synthesis

### Executive summary

As of 14 May 2026, the most defensible route to making OFARM ready for multi-agent AI and world-model AI is not a new AI authority plane and not a broad redesign. The standards landscape points in a narrower direction: workload identity and attestation for running software, delegated and revocable authority for each run, explicit capability discovery for tools and runtimes, provenance and policy-decision tracing for explainability, and credibility controls for digital-twin and forecasting outputs. These practices are spread across several mature ecosystems rather than one single ŌĆ£agent governanceŌĆØ standard: SPIFFE/SPIRE and the IETF WIMSE work for workload identity, OAuth token exchange and revocation plus OpenID Federation for delegated trust and cross-organisation metadata, MCP/OpenAPI/AsyncAPI/Kubernetes/FHIR for capability declaration, W3C PROV/OpenTelemetry/OPA for provenance and traceability, and NIST digital-twin and AI guidance for uncertainty, validation, and drift monitoring. That fragmentation supports a conservative OFARM amendment: keep baseline-law changes small and protective, and put most of the new machinery into RFCs, machine contracts, and conformance artefacts. The practical consequence is that OFARM needs a controlled amendment set with four emphases. First, protect existing law with a few baseline clarifications: no agent-held authority, no silent handoff of authority, no AI memory or projections as truth, no world-model state as Compliance state, and no tool-call success as a waiver of semantic law. ### Multi-agent governance patterns applicable to OFARM

#### Identity, sponsor, and authority

Current best practice is to separate **software identity**, **runtime instance identity**, **accountable sponsorship**, and **delegated authority**. SPIFFE/SPIRE and the IETF WIMSE workload-identity work treat running software as workloads that obtain cryptographically attested identities without long-lived embedded secrets. NIST zero-trust guidance treats non-person entities as first-class requesters whose access should never be trusted implicitly and should be continually evaluated. OAuth 2.0 Token Exchange adds the pattern for minting downstream, narrower credentials for specific calls, while Token Revocation adds the pattern for invalidating those credentials when a grant or sponsor state changes. OpenID Federation 1.1 adds signed entity metadata, trust chains, and federation policies for cross-organisation trust resolution. For OFARM, this implies a three-layer model: `SoftwareAgentProfile` as stable software identity, `AgentInstance` as ephemeral attested runtime identity, and `AgentSponsorRef` as the accountable farm, contractor, organisation, or natural person who sponsors the run. `AgentAuthorityEnvelope` should be the *only* carrier of allowed rights, and it should be sponsor-issued, time-bounded, purpose-bounded, revocable, non-transferable, and twin-scoped. ŅłĆciteŅłéturn39view3Ņłéturn39view4Ņłéturn40search0Ņłéturn40search1Ņłéturn39view1Ņłéturn39view2Ņłéturn39view0Ņłü

That pattern matters because identity answers ŌĆ£who is runningŌĆØ, while authority answers ŌĆ£what this run may do, in this context, for this sponsor, for this period, against these data and toolsŌĆØ. Combining those two concepts is exactly what creates silent privilege expansion in agent platforms. OFARM should therefore reject standing credentials, inherited blanket permissions, and ŌĆ£the agent is trusted because it is deployed inside our stackŌĆØ reasoning. Aligning with zero-trust and token-exchange practice, every run, and every downstream call within that run, should be evaluable against current sponsor status, revocation state, time window, purpose, sharing grant, and recipient tool policy. #### Agent run envelopes

A governed `AgentRunEnvelope` should be treated as a **transaction-scoped control document**, not as an execution log. The envelope should establish the legal and semantic perimeter for the run before execution starts. The most useful external patterns come from OpenTelemetryŌĆÖs agent span attributes, capability self-description in FHIR and Kubernetes, and tool declarations in MCP. At minimum, OFARM should require the envelope to carry: a stable `runId`; `SoftwareAgentProfile` and `AgentInstance` references; `AgentSponsorRef`; `AgentAuthorityEnvelope` reference and issuance chain; requested objective and user-visible purpose; declared twin scope; pack/profile compatibility references; `QuerySpecification` and `QueryPlanIR` references where relevant; approved data grants and redaction profile references; allowed tool IDs and tool-manifest digests; external-call policy; freshness and expiry requirements; required approval classes; allowed output-disposition classes; world-model policy flags; and trace/export requirements. It should also say what the run is *not* allowed to do, such as ŌĆ£no Compliance Twin writeŌĆØ, ŌĆ£no promotionŌĆØ, ŌĆ£no external internetŌĆØ, or ŌĆ£drafts onlyŌĆØ. ŅłĆciteŅłéturn27view0Ņłéturn27view1Ņłéturn18view3Ņłéturn22search0Ņłéturn19view5Ņłéturn19view1Ņłü

The envelope should also preserve OFARMŌĆÖs current-state and twin-separation law by forcing explicit declaration of whether the run may only read from governed materialisations, may propose drafts in the Advisory Twin, may create `BridgeCandidate` packages, or may only emit `EvidenceNeed` and `ObservationRequest` artefacts. Tool-call success should not imply semantic success, so the envelope should require separate reporting lanes for transport outcome, tool outcome, policy outcome, and OFARM-semantic outcome. That layered treatment mirrors FHIR `OperationOutcome`, which separates issue severity, issue code, details, and diagnostics instead of collapsing all failure into a single boolean. #### Agent run traces

A sufficient `AgentRunTrace` should explain behaviour rather than introspective reasoning. W3C PROV centres provenance on **entities**, **activities**, and **agents**, and its overview explicitly notes that provenance can refer to restricted resources by identifier even when the underlying content is not exposed. The PROV FAQ also explicitly recognises software agents as a subtype of agent. OPA decision logs show how policy queries can be logged with the queried policy, input, bundle metadata, and a `decision_id` that supports auditing and offline debugging. OpenTelemetryŌĆÖs GenAI and MCP semantic conventions add run roots, spans, links, tool-execution spans, MCP method attributes, and trace-context propagation. A practical OFARM trace should therefore expose a structured **read-set**, **query-set**, **assumption-set**, **tool-invocation-set**, **policy-decision-set**, **output-set**, **approval-set**, **blocked-action-set**, and **handoff-set**, with hashes and protected references where raw content is sensitive. ŅłĆciteŅłéturn14search0Ņłéturn14search5Ņłéturn18view5Ņłéturn27view0Ņłéturn27view1Ņłéturn27view2Ņłéturn28search0Ņłéturn28search1Ņłü

This is enough to answer the practical governance questions: what the agent read, what it assumed, what it called, what it produced, and what it was blocked from doing. It is *not* necessary, and is often unsafe or infeasible, to demand exposure of proprietary weights, hidden activations, or unrestricted chain-of-thought. NISTŌĆÖs explainable-AI principles point instead toward reasons that are evidence-backed, meaningful to users, and accurate to the process used, while model cards and system cards provide structured documentation of model capability, limitation, and evaluation context. In OFARM terms, the mandatory trace should be behavioural and externally verifiable; model internals can remain documented by reference through `AgentModelToolProfile` and linked model-card/system-card artefacts. #### Tool manifests and approval declaration

The strongest manifest pattern is **layered**, not monolithic. MCP provides direct tool metadata and warns that annotations are only hints; Kubernetes separates lightweight discovery from heavier OpenAPI schemas; OpenAPI provides machine-readable HTTP API descriptions; AsyncAPI does the same for message-driven interactions; Arazzo describes ordered call sequences and dependencies; FHIR `CapabilityStatement` distinguishes actual implementation capability from software capability and requirement profiles; ODRL expresses permissions, prohibitions, and duties; XACML adds explicit obligations and advice; and FHIR `OperationOutcome` provides a reusable issue structure for blocked or partially successful operations. OFARM should combine these ideas in `AgentToolManifest`: discovery fields, schema fields, side-effect class, open/closed world classification, approval class, required grants, declared obligations, refusal codes, and audit obligations should be separate concerns, not one blob of prose. ŅłĆciteŅłéturn19view5Ņłéturn19view1Ņłéturn20view0Ņłéturn18view3Ņłéturn24search15Ņłéturn25view0Ņłéturn24search4Ņłéturn22search0Ņłéturn35view4Ņłéturn38search0Ņłéturn18view4Ņłü

A particularly important lesson comes from MCP itself: `readOnlyHint`, `destructiveHint`, `idempotentHint`, and `openWorldHint` are explicitly described as *untrusted hints* and must not be the basis for trust-and-safety decisions unless they come from trusted servers. OFARM should adopt that exact posture. A manifest may declare that a tool is ŌĆ£read-mostlyŌĆØ, ŌĆ£externalŌĆØ, or ŌĆ£draft-producingŌĆØ, but the actual permission to call the tool, the approval it requires, and the semantic blockers that apply must all still be decided by OFARM policy and law. This prevents the classic failure mode where a tool self-describes as safe and the platform silently grants it authority. ŅłĆciteŅłéturn19view1Ņłéturn19view5Ņłü

A second useful lesson comes from AsyncAPI: it is **not recommended** to derive a sender document from a receiver document or vice versa, because directionality, purpose text, and infrastructure assumptions change. For OFARM, the analogue is that discoverability of a tool or event path must never imply reciprocal rights, write rights, or approval equivalence. Read capability does not imply promotion capability; advisory drafting does not imply Compliance Twin mutation; handoff receipt does not imply sponsor continuity. ŅłĆciteŅłéturn25view0Ņłü

#### Handoff without silent authority transfer

Task and context handoff should be explicit, signed, and narrow. The sender should be able to transfer a **context package**ŌĆöinput artefact references, open questions, assumptions still in force, blockers already encountered, and proposed next stepsŌĆöbut not reusable authority. RFC 8693 is designed around token *exchange* for delegation or impersonation, not raw forwarding of a bearer credential, and RFC 7009 exists precisely because delegated credentials need revocation semantics. That combination points to an OFARM `AgentHandoffEnvelope` with runtime provenance and requested next-scope information, but with a hard rule that the receiving agent must acquire its own `AgentAuthorityEnvelope` before taking any action. ŅłĆciteŅłéturn39view1Ņłéturn39view2Ņłü

The handoff object should therefore include `fromRunRef`, prior trace refs, context refs, assumption refs, requested next role, requested tool set, requested data grants, sponsor continuity status, expiry, and a mandatory ŌĆ£authority not transferredŌĆØ flag. If the recipient lacks a fresh sponsor-binding or if the sponsor has been revoked, the handoff may remain inspectable and useful, but it cannot authorise anything. In OFARMŌĆÖs terms, handoff may move **context** and may create a new Advisory artefact or BridgeCandidate package; it must never move **authority**. ## Twin and world-model findings

### World-model / digital-twin patterns applicable to OFARM

#### World-model state, scenario state, assumptions, and invalidation

NISTŌĆÖs current digital-twin work frames digital twins as forecasting-capable, lifecycle-connected, data-driven representations whose value depends on synchronisation with the physical counterpart, standardised interfaces, and ongoing validation. NISTŌĆÖs credibility paper says verification, validation, and uncertainty quantification must be applied across the digital-twin lifecycle, and NISTŌĆÖs trust work warns against treating the twin as an untestable oracle or gold standard. Agricultural digital-twin reviews point to high-value uses in crop monitoring, irrigation, cultivation support, and post-harvest processes, but they also describe an immature and fragmented field. OFARM should therefore support world models as **governed Advisory Twin artefacts**, not as a third twin and not as an alternate compliance state. ŅłĆciteŅłéturn29view0Ņłéturn29view1Ņłéturn30view0Ņłéturn29view3Ņłéturn32view0Ņłéturn31search5Ņłéturn15search1Ņłéturn15search9Ņłéturn15search13Ņłü

A credible `WorldModelState` should always answer seven questions. Which observations grounded it; which assumptions extended beyond those observations; which model and tool profile produced it; what uncertainty and intended-use statement applies; what validity window applies; what invalidation rules remove it from service; and how later observed outcomes will be reconciled back against the scenario result. That is the minimum contract if OFARM wants world models to remain useful but not dangerous. Invalidation rules should trigger on at least time expiry, stale or missing observations, model-version supersession, crop-stage transition, management actions outside scenario bounds, pack/profile incompatibility, or forecast/observation divergence above a declared threshold. Later reconciliation should create a new governed comparison artefactŌĆöuseful for calibration and readiness evidenceŌĆöbut not silently mutate Compliance Twin state or canonical history. ŅłĆciteŅłéturn32view0Ņłéturn12search8Ņłéturn7search8Ņłéturn29view0Ņłéturn29view3Ņłü

#### Advisory-to-authoritative boundaries

The best current practice is to keep prediction, recommendation, confidence, simulation, and memory in the **advisory** layer unless separate governed evidence supports promotion. NISTŌĆÖs explainable-AI guidance says AI systems should provide understandable and accurate reasons and should operate only under conditions for which they were designed, while the AI RMF playbook recommends regular monitoring for drift, decontextualisation, and negative impacts. NISTŌĆÖs trust considerations for digital twins specifically flag the danger of deciding that a digital twin is ŌĆ£the oracleŌĆØ or ŌĆ£gold standardŌĆØ. OFARM should therefore treat any `WorldModelRun` or agent recommendation as advisory by default, with `AgentOutputDisposition` values such as `simulation-only`, `advisory-draft`, `observation-request`, `bridge-candidate`, or `blocked`. None of those should ever be equal to accepted compliance fact. ŅłĆciteŅłéturn7search0Ņłéturn12search8Ņłéturn31search5Ņłü

That boundary should also cover ŌĆ£model memoryŌĆØ. Long-context chat history, vector stores, summarisation memory, and cached state can improve usability, but they must not become hidden truth stores. They are runtime aids, not canonical evidence. If a memory fragment matters to governance, it must be surfaced as an explicit OFARM artefact with provenance, evidence sufficiency checks, and governed promotion. Otherwise it remains non-authoritative execution residue inside the Advisory Twin or runtime boundary. ŅłĆciteŅłéturn14search0Ņłéturn27view0Ņłéturn27view2Ņłü

#### Evidence requests and observation requests

Useful AI-generated evidence requests should look like bounded work items, not unstructured nagging. FHIR `OperationOutcome` offers a useful shape for describing problems through severity, code, details, and diagnostics, while MCP annotations add a practical notion of intended audience, relative priority, and last-modified time. Participatory agricultural DSS work and FAO material on farmer knowledge both stress that advice becomes more usable when it is context-specific, co-designed around real conditions, and able to incorporate local knowledge instead of erasing it. OFARM should therefore define `EvidenceNeed` as the semantic statement of what is still missing and `ObservationRequest` as the farmer-facing or operator-facing task that could reduce uncertainty or unblock a governed decision. ŅłĆciteŅłéturn18view4Ņłéturn20view0Ņłéturn17search0Ņłéturn17search4Ņłéturn17search8Ņłéturn17search21Ņłü

The minimum useful `EvidenceNeed` fields are the blocked claim or decision, why current evidence is insufficient, decision sensitivity, acceptable evidence classes, and whether external evidence is mandatory before promotion. The minimum useful `ObservationRequest` fields are what to observe, where and when to observe it, acceptable collection methods, acceptable artefact forms, deadline, priority, burden estimate, safety note if any, option for local-knowledge annotation, and stop conditions to prevent repetitive or overwhelming requests. This pattern keeps AI in a support role while making the resulting workload actionable and proportionate for crop farms. ŅłĆciteŅłéturn18view4Ņłéturn17search0Ņłéturn17search4Ņłéturn17search21Ņłü

### Agricultural digital platform implications

The highest-value crop-farm scenarios are not generic ŌĆ£fully autonomous farm agentsŌĆØ. They are governed assistance patterns around evidence, planning horizon, selective disclosure, and reconstruction. FMIS literature shows that crop-farm systems already need to reduce production costs, support compliance with standards, and maintain product quality and safety. Agricultural digital-twin work points to irrigation, crop monitoring, cultivation support, and post-harvest planning. Resilience work in digital farming identifies interoperability gaps and internet dependency as continuing vulnerabilities, which is directly relevant for contractors, remote fields, and offline capture. Taken together, those sources suggest a compact scenario set in which OFARMŌĆÖs amendment would pay off first. ## Controlled change recommendations

### Patterns OFARM should reject

The common AI-agent platform patterns below are the ones most likely to weaken OFARMŌĆÖs existing law.

| Rejected pattern | Why OFARM should reject it |
|---|---|
| Standing all-powerful agents with reusable credentials | Collapses sponsor accountability, revocation, and least-privilege delegation |
| Agent memory, vector stores, or summaries treated as record | Creates hidden truth stores outside assertion/history-first governance |
| Auto-writeback from sensors, EO, FMIS imports, models, or tools | Treats external data or model outputs as accepted truth without governed promotion |
| A separate ŌĆ£AI TwinŌĆØ | Breaks the existing two-twin architecture and encourages authority leakage |
| Tool self-description as security approval | MCP explicitly warns that tool annotations are only hints and may be untrusted |
| Confidence scores as evidence sufficiency | Replaces OFARM evidence law with model self-confidence |
| Handoff that carries authority or approvals | Makes silent delegation and replay attacks likely |
| CRUD abstraction over mutable current-state tables | Flattens OFARM into CRUD and turns materialisations into truth stores |
| Packs or prompts that mutate core semantics | Undermines pack/profile merge law and semantic stability |
| Chat-log-only observability | Misses policy decisions, query execution, semantic blockers, and provenance links |
| Benchmarking or training by default on shared farm data | Weakens sovereignty, purpose limitation, and revocation discipline |
| Twin or model treated as oracle | NIST explicitly flags this as a trust failure pattern for digital twins |

These rejections are not anti-AI; they are anti-ambiguity. The external standards reviewed here repeatedly reward explicit declaration, separate capability and authority, machine-readable use restrictions, policy-decision IDs, and validation under uncertainty. OFARM should stay aligned with that direction. ŅłĆciteŅłéturn19view1Ņłéturn19view5Ņłéturn31search5Ņłéturn35view4Ņłéturn35view1Ņłéturn18view5Ņłü
