# OFARM2 Core

Status: implementation and conformance packaging profile — not OFARM law. Core = the smallest set of crop-farming semantics carried on the Kernel.

## Domain identities

Domain identities are Kernel `IdentityRecord`s plus typed **identity payloads** carried by structure assertions. The payload pattern: a `StructureEvent` + structure `AssertionRecord` carries the payload; the identity registry materializes the current payload per identity. Payload contracts (all **candidate**, in `contracts/core/`):

| Identity | Payload contract | Pilot notes |
|----------|------------------|-------------|
| Farm | `FarmIdentityPayload` | `holdingIdentifiers` carries scheme-bound national IDs. **KMG-MID is a Slovenia profile binding, not Core law.** |
| Field | `FieldIdentityPayload` | `parcelIdentifiers` carries scheme-bound parcel IDs (**GERK via SI profile**), with optional `referenceSnapshotRef` per identifier. Declared area is a quantity with a UCUM unit code. |
| CropCycle | `CropCycleIdentityPayload` | One cultivation attempt on a scope. `autoCreated: true` supports low-friction onboarding (created at first record, revisable later under identity-revision law). |
| Equipment | `EquipmentIdentityPayload` | The sprayer. `inspectionEvidenceRefs` carries mandatory equipment-test certificates as `EvidenceRecord` refs. |
| AppliedResource | `AppliedResourceIdentityPayload` | The product. Authorisation identity binds **only** through `AgronomicIdentityBinding` under the active code-binding profile. |

**Deferred identities** (defined in law, no payload contract cut in v1): Site, ManagementZone, MicroclimateZone, Lot, Facility, StorageLocation, Container. Cut them when a pilot flow needs them, not before.

**Deliberate deferral — `Operation`:** the Constitution lists Operation/OperationRecord as first-class, but the canonical contracts and fixtures route operations through `SemanticEventEnvelope` + `ExecutionRecordPayload` + `AcceptedEventConsequence`. Cutting a separate Operation identity contract now would create two representations of "an operation" before either is implementation-tested. v1 uses the event+payload chain; the Operation contract decision belongs to the post-pilot amendment. (Hostile-review finding F6.)

## The operation chain (how a spray application becomes truth)

1. **Capture** (offline-capable) produces a draft.
2. **Commit** via `CommitIngressRequest` (commit class: `operation claim`) creates an `InterventionEvent` (`SemanticEventEnvelope`) carrying an `ExecutionRecordPayload` (`recordClass` = operation claim; actor, extent, `actualAction`, `actualQuantityParameters`, equipment, `sourcePayload`, evidence refs, `promotionBoundary`).
3. **Gates** run: authority → validation (incl. carrier checks: quantity+unit, identity bindings, partial extent) → profile applicability → evidence sufficiency (auto-generated `EvidenceSufficiencyCase` against the profile floor) → review.
4. **Review** (`ReviewDecision`, per the SI profile's self-review policy) accepts → `AcceptedEventConsequence` (accepted executed intervention consequence).
5. **Materialization** updates Compliance-twin current state with a fresh `MaterializationBasis`.
6. Later correction or dispute = new `ExecutionRecordPayload` (`recordClass` correction/dispute) + supersession — never edits.

Plan-versus-execute (`InterventionIntentPayload`, `PlannedIntervention`) ships in contracts but has **no v1 UI** — the compliance record is execution-only; planning is a later product feature, not a law change.

## Observation and measurement

`NarrativeObservation` (free text preserved, never flattened), `AgronomicObservationContext` (structured context when high-consequence use needs it), `MeasurementEvidence` (quantity, unit, method, uncertainty), `PartialExtent` ("I only sprayed the west half" without inventing a new field). All extracted verbatim.

## Code binding discipline (the heart of Core)

- Crop names → EPPO codes. Growth stages → BBCH. Units → UCUM (quantity kinds → QUDT where needed). Products → the **jurisdictional product authorisation register** (FITO-INFO in the SI profile).
- Bindings are `AgronomicIdentityBinding` records: scheme + value + evidence + state. Unresolved bindings are explicitly `UNRESOLVED` — free text never silently becomes compliance identity.
- Registers enter only as dated `ReferenceSnapshot`s; verification against them is recorded as `ExternalRegistryVerificationTrace` (machine-generated by the registry adapter — see `CAPTURE_MAPPING.md`).
- Which schemes play which roles, with what evidence floors and unresolved-binding behavior, is declared per jurisdiction in an `AgronomicCodeBindingProfile` — see `profile_si_ffs/PROFILE.md`.

## Commit classes used in the pilot

`structure assertion` (identities), `operation claim` (spray records), `observation assertion`, `evidence record`, `note`, `advisory output` (e.g. the authorisation-mismatch warning), `governance decision` (reviews). `compliance assertion` and `hypothesis assertion` are available but unused in v1 flows.
