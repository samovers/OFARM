### B. Patterns directly applicable to OFARM

The best fit for OFARM is a **layered public contract pack** rather than a redesigned data model. OpenAPI 3.1 should describe synchronous HTTP surfaces; AsyncAPI should describe event channels and broker bindings for notifications and long-running job status; JSON Schema 2020-12 should hold reusable request, response, and envelope types; and a public overlay should strip internal-only operations and fields before publication. That is the smallest change that makes the runtime legible to AI agents while leaving OFARMŌĆÖs assertion/history-first truth model intact. It also fits OFARMŌĆÖs existing machine-contract posture much better than any attempt to invent a new agent-only fa├¦ade. For **capability discovery**, OFARM should borrow more from FHIR CapabilityStatement and OperationDefinition than from generic swagger dumps. CapabilityStatement exists specifically to declare actual system behaviour and can be used for compatibility testing, code generation, and conformance assessment; OperationDefinition includes whether an operation affects state, what scope it applies to, parameter shapes, and whether it profiles a base operation; SearchParameter and StructureDefinition show how codegen-facing names, derived forms, and computable constraints can be published with stable identifiers. The direct translation for OFARM is: every agent-callable operation should have a canonical operation contract, every query surface should advertise its supported parameters and query semantics, and every pack/profile addition should be additive and linked to base definitions rather than mutating them in place. ### F. Recommended public API surfaces

The public API should stay **semantic and governed**, not resource-CRUD and storage-shaped.

The first surface family is **discovery**: a capability endpoint, contract endpoints, schema endpoints, and reason-code registry endpoints. Concretely, something in the family of `GET /capabilities`, `GET /contracts/openapi`, `GET /contracts/asyncapi`, `GET /contracts/schemas/{schemaId}`, and `GET /contracts/problems` is enough. The second family is **query compilation and execution**, centred on OFARMŌĆÖs existing QuerySpecification and QueryPlanIR rather than on row-level resource mutation. Search/query parameter naming should be stable, kebab-cased, and documented machine-readably, following the same basic discipline that FHIR applies to SearchParameter names and processing modes. ŅłĆciteŅłéturn14search0Ņłéturn20search0Ņłéturn31view2Ņłü

The third family is **governed operation surfaces**. ## Delivery tooling

### G. Recommended SDK/codegen structure

The SDK shape should make the safe path the easiest path.

Generate **typed contract models**, **validators**, and **operation stubs** from the public contract pack only. ### H. Recommended conformance tests

The conformance suite should be aimed at **known agent failure modes**.

Start with **semantic-law blocker tests**: an agent must not be able to write current-state materialisations as truth; a projection must not be promotable as canonical evidence; an Advisory Twin result must not satisfy a Compliance Twin approval; a pack/profile addition must not change core semantic meaning; FMIS shadow imports must remain candidate-only until promotion. These are OFARM-specific tests and should be treated as hard blockers even if every HTTP contract test passes.

Add **contract-discipline tests**: unknown fields on public core objects must fail under strict validation; internal-only operations must not appear in published SDKs; public contracts must have stable canonical IDs and codegen-safe names; every non-2xx public response must map to a registered problem type and OFARM reason code; every high-consequence operation must advertise and honour dry-run, explain, and trace support if the capability manifest says it does. Kubernetes field validation and dry-run, FHIR codegen-oriented identifiers, and RFC 9457 problem typing all support those checks. ŅłĆciteŅłéturn3search1Ņłéturn30view1Ņłéturn1search1Ņłéturn20search0Ņłéturn14search0Ņłéturn24search0Ņłéturn5search1Ņłü

Add **sync and import tests**: replay the same import with the same idempotency key and verify a single semantic receipt; submit stale preconditions and expect 412 or 428 behaviour; provoke a real semantic conflict and require a 409 payload with merge-helpful detail; expire continuation cursors and require recoverable `410 Gone` behaviour; import unsupported units and require fidelity/loss-map warnings instead of silent coercion. ŅłĆciteŅłéturn17view1Ņłéturn18search0Ņłéturn31view2Ņłéturn31view3Ņłéturn11search4Ņłü

Add **numeric and display tests**: canonical unit conversions must match registered vectors; rounding must be stable and stage-specific; display rounding must never change governed results; stale/disputed/redacted/insufficient-evidence states must survive from API payload through SDK models into UI rendering. UCUM, FHIR Quantity, NIST conversion guidance, security labels, and Data Absent Reason support these checks. ŅłĆciteŅłéturn10search1Ņłéturn8search0Ņłéturn11search4Ņłéturn27search0Ņłéturn28search2Ņłü

Finally, add **explainability regression tests**: for a curated set of preflight and denial scenarios, the returned trace or explanation should match approved golden structures closely enough that SDKs and agents can rely on them. ## Governance gate

### I. Recommended agent-readiness gate

An OFARM runtime should not be declared agent-ready until all of these are true.

The first gate is **contract completeness**: the public contract pack exists, validates, is versioned, and is generated from authoritative OFARM machine contracts rather than hand-written after the fact. The public overlay produces a partner-safe contract set, and the released SDKs are generated from that same pack. ŅłĆciteŅłéturn29search3Ņłéturn22search2Ņłéturn2search1Ņłü

The second gate is **capability honesty**: the live capability manifest matches the runtime, every operation is explicitly marked allowed, approval-required, or forbidden for agents, and the manifest distinguishes advisory from authoritative behaviour. MCP-style hints may be present, but they are not treated as authority. ŅłĆciteŅłéturn4search0Ņłéturn4search1Ņłéturn1search1Ņłéturn20search0Ņłü

The third gate is **rehearsal safety**: every high-consequence operation supports preflight, returns structured reasons, and, where feasible, supports dry-run with real authorisation checks and zero side effects. Generated values that are nondeterministic in dry-run are clearly marked as such. ŅłĆciteŅłéturn30view1Ņłéturn30view2Ņłü

The fourth gate is **error determinism**: all public non-success outcomes map to registered problem types and OFARM codes; retry classes are declared; traces are retrievable when advertised; and SDKs can branch on codes and status rather than message text. ŅłĆciteŅłéturn5search1Ņłéturn1search0Ņłéturn19search0Ņłü

The fifth gate is **candidate-first ingestion**: every FMIS, machinery, or sensor adapter emits source fidelity, loss maps, unresolved identity candidates, and candidate-only receipts, and none can bypass governed promotion. ŅłĆciteŅłéturn12search0Ņłéturn12search1Ņłéturn27search0Ņłü

The sixth gate is **display honesty**: every public read surface emits a qualification envelope and all user-facing applications preserve stale, disputed, permission-limited, redacted, and evidence-insufficient states. ŅłĆciteŅłéturn27search0Ņłéturn28search2Ņłéturn28search0Ņłü

The seventh gate is **numeric determinism**: managed formulas, unit conversions, and rounding behaviours are versioned, tested, and uniform across server, SDK, and UI preview paths. ŅłĆciteŅłéturn10search1Ņłéturn8search0Ņłéturn11search4Ņłü

The eighth gate is **semantic-law conformance**: the OFARM-specific shortcut tests all pass, especially the ones that try to turn projections into truth, packages into semantic mutators, imports into accepted truth, or AI outputs into governance decisions. No amount of API polish should waive that gate.
